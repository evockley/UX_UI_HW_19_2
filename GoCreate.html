The Problem

Custom clothing salespeople have an antiquated order system that is cumbersome, confusing, and difficult to use. Contextual inquiry has shown that it can take as long as 90 minues to submit a single order. The frustrating experience effetively disincentivizes users from making sales in the department.

The Challenge

I wanted to retain the many details associated with custom clothing while allowing users to complete their tasks quickly, easily, and accurately.

The Solution

A highly organized graphic apprach to order input, designed with speed, ease, and mobile technology in mind.

User Flow

The User Flow was iterated based on feedback from users. My initial attempt at a user flow included fewer screens as that was one of the main pain points from users. However, I learned that the number of screens wasn‚Äôt really the issue. Users were happy to interact with more screens if there was less to do on each and fewer opportunities for error. In the fit tools module, there is more than one option per screen available, but when a user selects an individual alteration, an overlay screen makes it the only available interaction point. This approach tested well.

III. Ideation & Execution

Interviews revealed that users need more confidence in the system before they‚Äôre willing to trust it. It became imperative to design with ease, feeback, and error prevention in mind.

Design Goals

Make it as visual as possible

Shortcuts where possible

Consistent design patterns

Early Sketches

I hypothesized that I could eliminate much of the confusion of the existing interface by removing the screen clutter and making the experience more visual. I also wanted users to feel like they were in control of their own order.

Wireframes and Pivot

I ultimately moved forward with the search bar interface for a few reasons:

  • Users don‚Äôt need to interact with each dropdown on the Fit Tools page.
  • It was the most streamlined way to eliminate all the visual clutter from the page
  • It was the best way to show users only the alterations they wanted for a specific order

However, my first round of testing wasn‚Äôt met with a lot of enthusiasm from users. Between the time I began the project and this iteration, the company purchases iPads for each employee. Now, rather than going to a back office and entering orders, users were now expected to enter orders on their mobile devices.

This presented a new set of problems. In addition to the challenging interface, users needed to interact with it on a device with a smaller screen (no, it wasn‚Äôt responsive design), and with added disctractions to their workflow.

I needed to come up with a new solution.

Luckily, my sketches included something that sort of looked like tappable cards, so I adapted that approach for an iPad. Now, rather than relying on a searchbard (and therefore keystrokes), users were now able to tap plus and minus icons to increase or decrease a measurement. The most used alterations are on the first page, and users needing more obscure alteration can find them tucked away in tabs using standard industy language: Posture, Circumference, and Length.

This tested better, but I still wasn‚Äôt entirely happy with this approach‚Äîit still offered too many opportunities for error in a busy environment. I iterated further to remove the plus and minus icons from this level. In the revised version (below), a user taps into each alteration, where there are a large plus and minus button on either side of the image.
